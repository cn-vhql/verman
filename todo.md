# 本地文件版本管理工具（类似Git轻量版）需求规划

## 一、核心目标

打造一款**轻量、易用**的本地文件版本管理工具，支持文件夹绑定、文件变更监控、版本记录（含备注）、版本回滚，通过Tkinter提供简洁GUI，SQLite存储版本数据，无需复杂配置，聚焦“记录-回滚”核心场景。

## 二、核心需求（必须实现）

### 1. 项目管理（绑定文件夹）

- 「新建项目」：选择本地文件夹作为“工作区”，自动在该文件夹下创建隐藏数据库文件（`.verman.db`），关联工作区与存储，支持文件夹移动后仍可正常使用。
- 「打开项目」：选择已创建项目的工作区（识别 `.verman.db`），加载历史版本数据。
- 「关闭项目」：断开当前工作区连接，不影响文件本身。
- 「删除项目」：删除 `.verman.db`及所有版本记录（不删除工作区原始文件），需二次确认。

### 2. 变更监控

- 自动/手动扫描工作区文件（默认递归子文件夹），对比上一版本的文件**修改时间+哈希值（MD5/SHA1）**，识别三类变更：
  - 新增文件：当前工作区存在、上一版本无记录；
  - 修改文件：路径一致但哈希值变化；
  - 删除文件：上一版本有记录、当前工作区不存在。
- 变更列表实时展示：文件相对路径、变更类型（新增/修改/删除）、文件大小。

### 3. 版本记录

- 「提交版本」：用户确认变更后，输入版本备注（必填），系统自动生成版本号（按提交顺序递增，如 `v1.0`→`v1.1`，支持自定义版本号）。
- 版本元数据存储：版本号、提交时间（精确到秒）、备注、变更文件数量。
- 文件快照存储：每个版本的变更文件（新增/修改）存储完整二进制内容，删除文件仅记录“删除状态”（不存储内容），确保回滚时可恢复。

### 4. 版本回滚

- 「选择版本」：从版本列表中选中目标版本（显示版本号、提交时间、备注）。
- 「回滚操作」：
  - 回滚前可选“备份当前状态”（生成临时备份文件夹，避免误操作）；
  - 按目标版本的文件记录恢复：
    - 新增/修改文件：从SQLite读取二进制内容，写入工作区对应路径（自动创建父目录）；
    - 删除文件：删除工作区中该文件（若存在）；
  - 回滚后，当前工作区状态与目标版本完全一致，支持基于该状态继续提交新版本。

### 5. 版本查看

- 版本列表展示：左侧面板按“提交时间倒序”显示所有版本，包含核心信息（版本号、提交时间、备注、变更文件数）。
- 版本详情：选中某版本后，中间面板显示该版本的所有变更文件（相对路径、变更类型）。

## 三、扩展需求（可选实现，提升体验）

1. 「忽略文件」：支持在工作区创建 `.vermanignore`文件，按规则忽略无需监控的文件（如 `*.log`、`temp/`、`__pycache__/`），语法参考 `.gitignore`（仅支持简单通配符）。
2. 「版本导出」：将选中版本的所有文件导出到指定文件夹（避免覆盖当前工作区）。
3. 「版本对比」：选中两个版本，高亮显示文件差异（仅文件名/路径差异，暂不支持文件内容对比，降低复杂度）。
4. 「数据库备份」：手动导出 `.verman.db`到指定路径，支持后续恢复。
5. 「状态提示」：底部状态栏实时显示当前项目路径、变更文件数、数据库连接状态（正常/异常）。

## 四、非功能需求

1. 易用性：GUI操作步骤≤3步（如“打开项目→选版本→回滚”），无技术门槛。
2. 轻量性：无额外依赖（仅Python标准库+SQLite），数据库文件体积小（仅存储变更文件快照）。
3. 稳定性：
   - 提交版本时自动校验文件完整性，避免数据损坏；
   - 回滚前提示“是否备份当前状态”，防止误操作；
   - 异常退出时不破坏工作区文件和数据库。
4. 跨平台：支持Windows/macOS/Linux（依赖Python+Tkinter跨平台特性）。
5. 安全性：版本数据仅存储在本地，不涉及网络传输，保护文件隐私。

## 五、技术方案

### 1. 架构设计（三层架构）

```
GUI层（Tkinter）→ 业务逻辑层 → 数据存储层（SQLite）
```

- GUI层：负责用户交互（按钮、列表、输入框）；
- 业务逻辑层：处理项目绑定、变更检测、版本提交/回滚、哈希计算；
- 数据存储层：SQLite数据库，存储项目配置、版本元数据、文件快照。

### 2. 数据模型（SQLite表设计）

每个项目对应独立的 `.verman.db`，包含3张表：

#### （1）配置表（config）：存储项目基础信息

| 字段名 | 类型             | 说明                                                   |
| ------ | ---------------- | ------------------------------------------------------ |
| key    | TEXT PRIMARY KEY | 配置项标识（如project_path、create_time）              |
| value  | TEXT             | 配置值（如 `D:/work/test`、`2025-11-05 10:00:00`） |

#### （2）版本表（versions）：存储版本元数据

| 字段名       | 类型                              | 说明                  |
| ------------ | --------------------------------- | --------------------- |
| id           | INTEGER PRIMARY KEY AUTOINCREMENT | 版本唯一ID（自增）    |
| version_num  | TEXT NOT NULL                     | 版本号（如 `v1.2`） |
| commit_time  | DATETIME NOT NULL                 | 提交时间（ISO格式）   |
| comment      | TEXT                              | 版本备注（允许为空）  |
| change_count | INTEGER NOT NULL                  | 该版本变更文件总数    |

#### （3）文件表（files）：存储版本对应的文件快照

| 字段名                                                             | 类型                              | 说明                                          |
| ------------------------------------------------------------------ | --------------------------------- | --------------------------------------------- |
| id                                                                 | INTEGER PRIMARY KEY AUTOINCREMENT | 文件记录唯一ID                                |
| version_id                                                         | INTEGER NOT NULL                  | 关联versions表的id（外键）                    |
| relative_path                                                      | TEXT NOT NULL                     | 文件相对工作区的路径（如 `doc/readme.txt`） |
| file_hash                                                          | TEXT NOT NULL                     | 文件哈希值（MD5）                             |
| file_status                                                        | TEXT NOT NULL                     | 变更类型（`add`/`modify`/`delete`）     |
| file_content                                                       | BLOB                              | 文件二进制内容（`delete`状态时为空）        |
| FOREIGN KEY (version_id) REFERENCES versions(id) ON DELETE CASCADE |                                   | 版本删除时同步删除关联文件记录                |

### 3. GUI设计（简洁布局）

#### 整体结构（自上而下）

1. 菜单栏（顶部）

   - 项目：新建/打开/关闭/删除/退出
   - 版本：提交/回滚/刷新变更/导出版本
   - 工具：忽略文件配置/数据库备份
   - 帮助：使用说明/关于
2. 主面板（中间，分三列）

   - 左侧：版本列表（滚动条），列名：版本号、提交时间、备注、变更数
   - 中间：文件变更列表（滚动条），列名：相对路径、变更类型、文件大小
   - 右侧：操作区
     - 备注输入框（提交版本时启用，多行文本）
     - 功能按钮：【刷新变更】【提交版本】【回滚选中版本】【导出版本】
     - 状态提示：当前变更统计（如“3个文件变更：1新增+2修改”）
3. 状态栏（底部）

   - 左侧：当前项目路径（如 `D:/work/test`）
   - 中间：数据库状态（如“已连接”“未连接”）
   - 右侧：当前时间

#### 关键交互逻辑

- 未打开项目时：仅“新建项目”“打开项目”“退出”可用，其他按钮置灰；
- 打开项目后：自动扫描首次变更，显示在变更列表；
- 提交版本：需填写备注（否则提示），提交后清空变更列表，版本列表新增记录；
- 回滚版本：选中版本后按钮激活，点击后弹出“备份提示”，回滚完成后刷新变更列表（显示为空）。

## 六、非功能约束

1. 支持文件类型：无限制（但建议仅用于中小文件，大文件（>100MB）可能导致数据库体积过大、操作卡顿）；
2. 性能：单次扫描文件数≤1000个（满足日常小项目需求），版本提交/回滚响应时间≤3秒；
3. 兼容性：Python 3.8+（Tkinter内置，SQLite无需额外安装）；
4. 安全性：删除项目/回滚操作需二次确认，避免误删数据。

## 七、开发步骤（按优先级排序）

1. 搭建基础架构：SQLite表创建（config/versions/files）、项目绑定（新建/打开/关闭）；
2. 实现变更检测逻辑：文件扫描、哈希计算、变更识别与展示；
3. 实现版本提交：备注输入、版本号生成、文件快照存储；
4. 实现版本回滚：版本数据读取、文件恢复、备份逻辑；
5. 开发GUI核心界面：菜单栏、版本列表、变更列表、操作区；
6. 完善交互细节：状态提示、按钮置灰、二次确认弹窗；
7. 开发扩展功能：忽略文件、版本导出、数据库备份；
8. 测试与优化：边界测试（空文件夹、大文件、多版本回滚）、性能优化、bug修复。

## 八、使用流程示例

1. 打开工具 → 新建项目 → 选择 `D:/work/test`作为工作区；
2. 在 `test`文件夹中创建/修改文件（如 `a.txt`）；
3. 工具自动扫描到“1个新增文件”，在变更列表显示；
4. 输入备注“初始化a.txt文件”，点击【提交版本】，生成 `v1.0`；
5. 后续修改 `a.txt` → 工具识别“1个修改文件” → 提交 `v1.1`（备注“更新a.txt内容”）；
6. 如需回滚：在版本列表选中 `v1.0` → 点击【回滚选中版本】 → 确认备份 → 工作区 `a.txt`恢复为 `v1.0`状态。
